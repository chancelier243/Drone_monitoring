1. INTRODUCTION & OBJECTIF
================================================================================

L'application "Drone Monitoring" est une application mobile et multi-plateforme
créée avec Flutter. Elle permet de monitorer en temps réel les données de
télémétrie d'un drone (altitude, vitesse, angles, température, batterie, etc)
grâce à une connexion HTTP avec un serveur backend.

OBJECTIF PRINCIPAL:
  → Visualiser les données du drone en direct (dashboard temps réel)
  → Analyser l'historique avec des graphiques dynamiques
  → Consulter les données brutes dans un tableau historique
  → Configurer la connexion au serveur depuis l'interface

PLATEFORME SUPPORTÉES:
  ✓ Android (4.6+ / API 21+)
  ✓ iOS (11.0+)
  ✓ Windows (10+)
  ✓ Linux (GTK 3.22+)
  ✓ macOS (10.11+)
  ✓ Web (navigateurs modernes)

4.2 LIB/DATA_MODEL.DART - Modèle de données
───────────────────────────────────────────────────────────────────────────────

Classe: TelemetryData

Attributs (11 champs):
  • timestamp      DateTime    → Horodatage de la mesure
  • pitch          double      → Inclinaison avant/arrière (degrés) [-90, 90]
  • roll           double      → Inclinaison gauche/droite (degrés) [-180, 180]
  • yaw            double      → Orientation (degrés) [0, 360]
  • altitude       double      → Hauteur au-dessus du sol (mètres)
  • speed          double      → Vitesse horizontale (m/s)
  • battery        double      → Tension batterie (volts)
  • temperature    double      → Température du moteur/batterie (°C)
  • acceleration   double      → Accélération (m/s²)
  • pressure       double      → Pression atmosphérique (hPa)
  • alerts         List<String> → Messages d'alerte (ex: "BATTERIE FAIBLE")

Méthodes:
  • Constructeur personnalisé avec valeurs par défaut
  • Factory fromJson() :
    - Parse le JSON reçu du serveur
    - Gère les variantes de noms (vitesse/speed, batterie/battery, etc)
    - Récupère timestamp actuel si absent
    - Récupère l'accélération et pression par défaut si absentes
    - Parse la liste des alertes

Exemple JSON attendu du serveur:
  {
    "timestamp": "2026-04-10T14:30:45.123Z",
    "pitch": 5.2,
    "roll": -3.1,
    "yaw": 180.0,
    "altitude": 45.5,
    "speed": 15.3,
    "battery": 12.4,
    "temperature": 32.5,
    "acceleration": 2.1,
    "pressure": 1013.25,
    "alerts": ["VITESSE ÉLEVÉE"]
  }


4.3 LIB/TELEMETRY_SERVICE.DART - Service de connexion
───────────────────────────────────────────────────────────────────────────────

Classe: TelemetryService (extends ChangeNotifier)

Rôle: Gérer la connexion HTTP, récupérer les données, maintenir l'historique

Attributs:
  • currentData      TelemetryData? → Dernière mesure reçue
  • history          List<TelemetryData> → Max 100 points pour graphiques
  • serverUrl        String → "http://localhost:5000" (configurable)
  • isConnected      bool → État de connexion (vert/rouge sur UI)
  • errorMessage     String → Message d'erreur détaillé si problème
  • failedAttempts   int → Compteur de tentatives échouées
  • _timer           Timer → Récupération toutes les 1 seconde

Méthodes principales:

  1. setServerUrl(String url)
     - Valide et nettoie l'URL passée en paramètre
     - Ajoute "http://" si absent
     - Retire le slash final
     - Réinitialise les compteurs d'erreur
     - Lance immédiatement une tentative de connexion

  2. startFetching()
     - Démarre un Timer qui appelle fetchTelemetry() chaque seconde
     - Appelé dans le constructeur

  3. fetchTelemetry() [ASYNC]
     - Effectue une requête GET sur {serverUrl}/latest-data
     - Headers: 'Content-Type': 'application/json'
     - Timeout: 8 secondes
     - Gestion des réponses:
       * 200 OK: Parse JSON, met à jour currentData et history
       * 401/403: Accès refusé, erreur affichée
       * Autres: Error {statusCode}
     
     - Gestion des exceptions:
       * TimeoutException → Message timeout
       * FormatException → Réponse invalide (pas du JSON)
       * SocketException → Serveur injoignable
       * HandshakeException → Erreur SSL/HTTPS
     
     - À chaque changement: notifyListeners() pour rafraîchir l'UI

Particularité: Support de localtunnel (loca.lt)
  - Header spécial 'bypass-tunnel-reminder' pour contourner page confirmation
  - Utile pour tests distants sans serveur dédié

4.7 SERVER.PY - Serveur backend
───────────────────────────────────────────────────────────────────────────────

Framework: Flask + Flask-CORS

Démarrage:
  python server.py

Écoute sur:
  http://0.0.0.0:5000

Routes disponibles:

1. GET /latest-data
   Retourne:
   {
     "pitch": 0.0,
     "roll": 0.0,
     "yaw": 0.0,
     "altitude": 0.0,
     "speed": 0.0,
     "battery": 12.0,
     "temperature": 25.0,
     "acceleration": 0.0,
     "pressure": 1013.25,
     "alerts": []
   }

2. POST /drone-data
   Reçoit les données du drone depuis une source externe
   Met à jour latest_telemetry avec les champs fournis
   Affiche dans terminal: [RECU] Alt=... Vit=... Bat=...
   
3. GET|POST /dashboard/telemetry
   GET: Retourne latest_telemetry
   POST: Reçoit et met à jour latest_telemetry
   Compatible avec Mission Planner

4. GET /health
   Retourne:
   {
     "status": "ok",
     "drone_connected": (altitude > 0)
   }
   Utile pour vérifier serveur en ligne

CORS activé: Permet requêtes depuis n'importe quelle origine
  Accepte les requêtes depuis localhost, réseau local, internet, etc.

Données serveur (dictionnaire global latest_telemetry):
  - Initialisées avec valeurs par défaut
  - Mises à jour via POST /drone-data
  - Retournées via GET /latest-data
  - Persistent en mémoire uniquement (pas de base de données)

5.2 CYCLE DE MISE À JOUR EN DIRECT (répétition chaque seconde)
───────────────────────────────────────────────────────────────────────────────

1. Timer déclenche fetchTelemetry()

2. fetchTelemetry():
   GET http://localhost:5000/latest-data
   ↓
   (Serveur répond avec JSON latest_telemetry)
   ↓
   App parse le JSON en TelemetryData
   ↓
   currentData ← nouvelle mesure
   ↓
   history.add(nouvelle mesure)
   ↓
   Si history.length > 100: supprimer le plus ancien
   ↓
   isConnected = true
   errorMessage = ""
   ↓
   notifyListeners() → Rafraîchit tout l'UI

3. UI se rafraîchit (Consumer<TelemetryService>):
   - Dashboard voit currentData actualisé → affiche nouvelles valeurs
   - Graphiques voient history actualisée → redessinent courbes
   - Tableau voit history actualisée → ajoute nouvelle ligne

Ceci se répète toutes les secondes ≈ 60 mises à jour/minute

4. _pages[1] → ChartsPage() affichée
5. ChartsPage accède à service.history → trace graphique

7. Chaque nouvelle mesure: graphique se met à jour en live


5.4 RECONFIGURATION URL SERVEUR
───────────────────────────────────────────────────────────────────────────────

Utilisateur clique sur ⚙️ Settings:

1. Dialog s'ouvre avec champ texte pré-rempli (URL actuelle)

2. Utilisateur modifie l'URL
   Exemples:
   - localhost:5000
   - 192.168.1.15:8080
   - https://my-tunnel.loca.lt

3. Clique sur "Connexion":
   - setServerUrl() appelé
   - URL nettoyée (ajout http:// si nécessaire)
   - Slash final retiré
   - failedAttempts ← 0
   - fetchTelemetry() lancé immédiatement
   
4. Résultat:
   - ✓ Connecté: currentData mis à jour avec nouvelles données
   - ✗ Non: errorMessage affiché (raison du refus)

5. La connexion s'essaie toutes les 1s (timer continue)
   Si succès: UI passe au dashboard avec données


5.5 GESTION DES ERREURS RÉSEAU
───────────────────────────────────────────────────────────────────────────────

Scénarios d'erreur:

1. Serveur injoignable (ne répond pas):
   → SocketException / Connection refused
   → Message: "Serveur injoignable. Vérifiez l'URL et que le serveur tourne."
   → isConnected = false
   → failedAttempts++
   → Retry automatique chaque seconde

2. Timeout (prend trop longtemps):
   → TimeoutException après 8 secondes
   → Message: "Connexion expiree (timeout 8s)"
   → isConnected = false
   → failedAttempts++

3. Réponse invalide (pas du JSON):
   → FormatException
   → Message: "Réponse invalide (pas du JSON). Vérifiez l'URL."
   → Exemple: Server retourne HTML au lieu de JSON
   → isConnected = false

4. Erreur SSL/HTTPS:
   → HandshakeException / CERTIFICATE
   → Message: "Erreur SSL/HTTPS. Essayez avec http:// à la place."
   → isConnected = false

5. Accès refusé (401 / 403):
   → Message: "Accès refusé ({statusCode}). Vérifiez l'URL."
   → isConnected = false

Tous les cas:
  • UI affiche message d'erreur en rouge
  • Bouton Settings reste disponible
  • Retry automatique continue chaque seconde

9. STRUCTURE DES DONNÉES ÉCHANGÉES
================================================================================

FORMAT JSON (dans les deux sens):

Depuis serveur vers app (GET /latest-data):
─────────────────────────────────────────────────────────────────────────────

{
  "timestamp": "2026-04-10T14:30:45.123Z",
  "pitch": 5.2,
  "roll": -3.1,
  "yaw": 180.0,
  "altitude": 45.5,
  "speed": 15.3,
  "vitesse": 15.3,                      ← Alternative (support français)
  "battery": 12.4,
  "batterie": 12.4,                     ← Alternative (support français)
  "temperature": 32.5,
  "acceleration": 2.1,
  "accel": 2.1,                         ← Alternative
  "pressure": 1013.25,
  "pression": 1013.25,                  ← Alternative
  "alerts": [
    "VITESSE ÉLEVÉE",
    "BATTERIE FAIBLE"
  ]
}

Champs requis:        pitch, roll, yaw, altitude, speed, battery, temperature
Champs optionnels:    timestamp, acceleration, pressure, alerts
Variantes acceptées:  vitesse/speed, batterie/battery, accel/acceleration

Depuis app vers serveur (POST /drone-data):
─────────────────────────────────────────────────────────────────────────────

Même format. Tout champ envoyé met à jour latest_telemetry sur serveur.


TYPES DE DONNÉES:
─────────────────────────────────────────────────────────────────────────────

Angle (pitch, roll, yaw):
  Plage acceptable: [-360, 360] (en pratique: [-90,90], [-180,180], [0,360])
  Unité: degrés
  Précision: 0.1°

Altitude:
  Plage: [0, ... ∞]
  Unité: mètres
  Précision: 0.1m

Vitesse:
  Plage: [0, ...]
  Unité: m/s
  Précision: 0.1 m/s

Batterie:
  Plage: [0, 25]
  Unité: volts
  Précision: 0.1 V
  Alerte: < 10.8V = BATTERIE FAIBLE

Température:
  Plage: [-40, 85]
  Unité: Celsius
  Précision: 0.5°C
  Alerte: > 60°C = SURCHAUFFE

Accélération:
  Plage: [-16, 16]
  Unité: m/s²
  Précision: 0.1 m/s²

Pression:
  Plage: [300, 1100]
  Unité: hPa
  Précision: 0.1 hPa

Alerte (alert string):
  Exemples: "BATTERIE FAIBLE", "SURCHAUFFE", "VITESSE ÉLEVÉE"
  Liste libre (n'importe quelle chaîne acceptée)

10. GUIDE DE DÉPANNAGE
================================================================================

Problème: App affiche "Serveur injoignable"
──────────────────────────────────────────────────────────────────────────────
Solutions possibles:
  1. Vérifier serveur lancé: python server.py → regarde message [OK]
  2. Vérifier port: localhost:5000 accessible (pas de firewall)
  3. Vérifier URL dans app: ⚙️ → "http://localhost:5000" exact
  4. Sur réseau: utilisez IP 192.168.x.x au lieu de localhost
  5. Relancer app: fermer et flutter run


Problème: "Réponse invalide (pas du JSON)"
──────────────────────────────────────────────────────────────────────────────
Causes possibles:
  1. Serveur retourne HTML (ex: page d'erreur)
  2. URL incorrecte (apprend page web au lieu de API)
  3. localtunnel: doit visiter URL une fois dans navigateur
  
Solutions:
  1. Tester avec curl: curl http://localhost:5000/latest-data
  2. Vérifier réponse: doit être JSON valide
  3. Si localtunnel: visiter https://XX.loca.lt dans navigateur d'abord
  4. Ajouter header dans app:
     headers['bypass-tunnel-reminder'] = 'true' ← Déjà fait!


Problème: App crée blanche au lancement
──────────────────────────────────────────────────────────────────────────────
Solutions:
  1. Attendre quelques secondes (premier lancement lent)
  2. Vérifier terminal Flutter: chercher erreurs/exceptions
  3. flutter clean && flutter pub get && flutter run
  4. Vérifier dépendances: flutter pub get


Problème: Graphiques ne s'affichent pas
──────────────────────────────────────────────────────────────────────────────
Causes:
  1. Pas encore 2 points de données (besoin min 2 pour graphique)
  2. Erreur de parsing JSON
  3. server.py ne correct pas de données
  
Solutions:
  1. Lancer test_with_mock_data.ps1 pour générer données
  2. Attendre 30 secondes pour accumulation points
  3. Vérifier terminal Python pour erreurs


Problème: App freeze ou lente
──────────────────────────────────────────────────────────────────────────────
Causes possibles:
  1. Trop d'historique (> 10000 points)
  2. Serveurnétions lentes
  3. Appareil lent
  
Solutions:
  1. Redémarrer app (vide historique)
  2. Réduire fréquence: modifier timer 1s → 2s
  3. Vérifier performance serveur


Problème: Connection timeout même avec serveur lancé
──────────────────────────────────────────────────────────────────────────────
Causes:
  1. Serveur ne respond pas
  2. Firewall bloque port 5000
  3. Adresse IP incorrecte
  
Solutions:
  1. Vérifier serveur: python server.py affiche "Serveur lancé"
  2. Test direct: Ouvrir http://localhost:5000/latest-data dans navigateur
  3. Vérifier firewall: Autoriser Python port 5000
  4. Sur réseau: Utiliser ifconfig / ipconfig pour trouver IP


Problème: Données ne se mettent pas à jour
──────────────────────────────────────────────────────────────────────────────
Causes:
  1. Pas de script de test en cours
  2. currentData toujours null
  3. Erreur de parsing
  
Solutions:
  1. Lancer: .\test_with_mock_data.ps1 ou python test_drone_data.py
  2. Envoyer données manuellement avec curl:
     curl -X POST http://localhost:5000/drone-data -H "Content-Type: application/json" -d '{"altitude": 50.5, "speed": 10.2}'
  3. Vérifier terminal app pour exceptions

12. LIMITATIONS & CONSIDÉRATIONS
================================================================================

Limitations actuelles:
───────────────────────────────────────────────────────────────────────────────

  • Historique en mémoire seulement (100 points conservés)
    → Données perdues à redémarrage

  • Pas de base de données persistante
    → Solution: Implémenter SQLite ou Firebase

  • Pas de chiffrement HTTPS (HTTP par défaut)
    → Production: Ajouter certificats SSL/TLS

  • Pas d'authentification/login
    → Production: Ajouter JWT ou OAuth2

  • Une seule connexion client acceptée
    → Production: Gérer multiple clients simultanés

  • Pas de notifications push
    → Pour critique: Implémenter alertes

  • Pas de sauvegarde/export données
    → À-faire: CSV, PDF, nuage

  • Pas de Mode hors-ligne
    → À-faire: Cache local avec synchronisation


Considérations de performance:
───────────────────────────────────────────────────────────────────────────────

  • Grande fréquence réseau (1 req/sec)
    → Peut surcharger sur réseau très lent
    → Solution: Augmenter intervalle Timer

  • Stockage graphiques (100 points par onglet)
    → Néglible pour FL Chart
    → Pas de problème de mémoire

  • Affichage 3D Horizon artificiel
    → Vérifier performance sur appareils anciens
    → Solution: Réduire detailDetail de rendu


Considérations de sécurité:
───────────────────────────────────────────────────────────────────────────────

  • CORS ouverte à toutes origines
    → Production: Restreindre domaines autorisés

  • Données non chiffrées en transit
    → Production: Implémenter HTTPS obligatoire

  • Pas de validation strict du JSON
    → Risque injection: minimiser par type casting

  • Pas de rate limiting
    → Production: Ajouter throttling des requêtes

  • Pas de logs persistantes
    → Production: Ajouter logging sécurisé
